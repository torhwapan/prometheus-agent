# AI + Prometheus Agent 的价值定位与优势场景

本文档用于说明：在已经具备 `Prometheus + Grafana` 的前提下，引入 `AI + Prometheus Agent` 的价值到底在哪里，以及它更适合解决哪些常用场景中的问题。

## 1. 先说结论

`Prometheus + Grafana` 擅长的是：

- 指标采集
- 指标展示
- 固定规则告警
- 告警通知

`AI + Prometheus Agent` 擅长的是：

- 巡检
- 风险前置发现
- 问题解释
- 多指标关联分析
- 自然语言查询
- 面向领导或非专家的结果输出

所以，二者不是替代关系，而是分工关系：

- `Prometheus`：负责采集与存储
- `Grafana`：负责看板与实时告警
- `Prometheus Agent`：负责巡检、预测、解释、归因、报告

一句话概括：

- `Grafana` 更擅长发现“已经发生的问题”
- `Prometheus Agent` 更擅长提前发现“正在形成的风险”

## 2. 为什么不能只靠 Grafana

如果只看监控系统能力，Grafana 已经足够强大。

但在实际使用中，仍然有几个明显短板：

1. 能看到图，但不一定能快速理解图
2. 能配置阈值，但对趋势型风险支持有限
3. 能做单指标告警，但不擅长多指标关联解释
4. 能展示数据，但不擅长自动形成巡检结论
5. 能让熟悉监控的人使用，但不适合非专家直接提问

也就是说，Grafana 更偏“监控系统能力”，而 Agent 更偏“把监控结果变成可消费的信息”。

## 3. AI + Prometheus Agent 的核心优势

### 3.1 每日巡检与汇报能力更强

Grafana 能做 dashboard，但不擅长直接生成一份适合汇报的巡检报告。

Agent 更适合自动输出：

- 哪些 job / instance 风险最高
- 哪些指标已经出现危险趋势
- 风险等级是什么
- 预计多久会触发上限
- 建议先关注哪些系统

这类输出更适合：

- 每日巡检
- 晨报 / 周报
- 领导汇报
- 稳定性例会

### 3.2 更适合做“提前发现风险”

Grafana 的主要优势在于阈值告警，即：

- 指标已经过线
- 条件已经满足
- 系统马上通知人

但很多运维工作真正想做的是：

- 还没报警，但趋势很危险
- 当前值正常，但增长速度不正常
- 现在没事，但预计几小时后会撞线

Agent 在这里更有价值，因为它可以结合：

- 环比分析
- 突变分析
- 持续增长分析
- 时间到上限预测

这类能力，更适合“提前引发关注”，而不是“事后通知”。

### 3.3 更适合处理弱规则和业务指标

很多指标并不适合写死阈值，例如：

- 白天高、晚上低
- 有周期波动
- 发版后短期抬升
- 大促期间整体上扬
- 某些业务指标本身没有统一阈值

Grafana 对这类指标通常还是依赖人工看图，或者配置比较粗糙的静态阈值。

Agent 更适合做：

- 和历史相比是否异常
- 与自身过去一段时间相比是否突然放大
- 是否持续增长且没有回落
- 是否偏离正常基线

尤其对自定义业务指标，这一点很有价值。

### 3.4 更适合做多指标关联分析

真实问题通常不会只表现为一个指标异常，而是一串信号同时变坏。

例如：

- JVM heap 使用率上升
- GC 时间升高
- 线程数增加
- Redis 连接数升高
- RabbitMQ 消费利用率下降

Grafana 能把这些图画出来，但不会自动告诉你：

- 这些异常是否可能属于同一个问题
- 哪一个是主因，哪一些是伴随现象
- 应该先排查哪个方向

Agent 可以把这些指标聚合后交给 AI 做关联解释，从而输出更像“巡检结论”的内容。

### 3.5 更适合自然语言问诊

Grafana 的使用方式是：

- 找 dashboard
- 选变量
- 切时间范围
- 看图
- 自己总结

Agent 的使用方式可以是：

- “帮我看下 10.22.23.24 上的 redis 有没有异常”
- “巡检一下 rabbitmq 哪些队列消费能力不足”
- “看看最近 24 小时 JVM 是否有明显内存风险”

也就是说，Agent 把“会不会看监控图”转成了“会不会提问题”。

这能显著降低使用门槛，扩大使用人群。

### 3.6 更适合给非专家使用

并不是每个使用监控系统的人都擅长：

- 写 PromQL
- 识别指标名
- 理解导出器语义
- 从多张图中拼出结论

对这些角色来说，Agent 更友好：

- 运维值班人员
- 应用开发人员
- 测试人员
- 技术负责人
- 业务负责人

他们不需要知道所有底层指标细节，也能拿到相对清晰的结论。

## 4. 最常见、最值得落地的场景

如果要让这个项目真正体现价值，最推荐优先落地下面几个场景。

### 4.1 自动巡检报告

每天定时巡检多个 Prometheus 服务器，输出 HTML 报告，重点列出：

- 高风险 job / instance
- 指标风险等级
- 风险原因
- AI 简评
- AI 关联分析

这是最容易被管理层和运维负责人直接感知到价值的场景。

### 4.2 对话式专项巡检

用户通过聊天框直接提问，Agent 自动：

- 识别目标 Prometheus
- 识别目标 job / instance
- 选择指标
- 查询数据
- 输出结论

这个场景适合临时排查和日常使用。

### 4.3 告警前的趋势预警

当指标尚未越过 Grafana 告警阈值时，Agent 可以通过趋势分析发现：

- 是否在快速逼近风险区
- 是否已经持续恶化
- 是否值得提前人工关注

这正好补足 Grafana 更偏“过线告警”的能力边界。

### 4.4 自定义业务指标巡检

很多业务指标没有成熟阈值体系，或者监控语义只有业务方自己清楚。

这类场景更适合通过：

- 指标分类 skill
- 指标语义说明
- 自定义 prompt
- AI 解释

来提升巡检效果。

## 5. 与 Grafana 的边界应该怎么划

为了避免系统职责重复，推荐这样划分：

### 5.1 Grafana 继续负责

- 实时看板
- 固定阈值告警
- 告警通知和路由
- 值班时的快速观测

### 5.2 Agent 重点负责

- 定时巡检
- 预测性分析
- 多指标归并
- AI 解释
- 自然语言查询
- HTML / Markdown 汇报输出

这样分工最合理，也最容易对内讲清楚价值。

## 6. 什么时候 Agent 价值不大

这点也要明确，否则容易把系统做重。

如果只是做下面这些事情，Agent 的价值并不明显：

- 单纯展示监控图
- 单纯替代 Grafana 阈值告警
- 单纯返回 Prometheus 原始查询结果
- 不做趋势分析、不做解释、不做关联、不做报告

如果 Agent 只是在“把 Grafana 已经会做的事情再做一遍”，那就很难体现差异。

所以，Agent 的重点不应放在“再次告警”，而应放在：

- 提前发现风险
- 自动形成结论
- 降低理解成本
- 提高巡检效率

## 7. 对内汇报时可以怎么概括

如果要用一句比较适合内部沟通的话来描述：

> Prometheus + Grafana 解决的是“系统是否看见了问题”，  
> AI + Prometheus Agent 解决的是“人是否能更早、更快、更低门槛地理解风险并采取行动”。

也可以换一种更偏落地的话术：

- Grafana 负责“监控和告警”
- Agent 负责“巡检和解释”

## 8. 最终建议

这个项目最值得坚持的方向，不是做成另一套告警平台，而是做成一套：

- 面向巡检的
- 面向趋势风险的
- 面向报告输出的
- 面向自然语言使用的
- 面向非专家也能消费的

Prometheus 智能巡检能力。

如果后续继续演进，最值得优先增强的方向通常是：

1. 指标分类与指标字典
2. 更稳定的趋势分析规则
3. 业务指标的基线和语义配置
4. AI 关联分析质量
5. 更适合汇报的报告模板
