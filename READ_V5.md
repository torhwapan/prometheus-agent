# Prometheus Agent V5 设计说明

## 1. 定位

V5 不再定义成一套新的 Python 服务。

V5 的定位是：

- 一组给公司 AI 平台使用的提示词
- 一组约束自然语言问诊行为的 skill
- 用于把用户临时提问稳定地转成结构化意图

也就是说，V5 只负责两件事：

1. 理解用户问题
2. 规范模型输出

真正的查询、分析、筛选、报表生成，继续复用现有的 Python 执行层能力，例如：

- V4 的固定巡检能力
- 你们后续新增的 Prometheus 查询工具
- 你们后续新增的条件筛选工具

这样拆分更合适，因为你们平台本身就负责：

- 串联大模型
- 调用工具
- 编排执行顺序

V5 没必要再额外做一层独立服务。

## 2. 目录结构

当前 V5 建议保持成下面这样的资产目录：

```text
prompts_v5/
  README.md
  intent_extraction.md
  result_explanation.md
  skills/
    prometheus-v5-dialog/
      SKILL.md

READ_V5.md
```

说明：

- `prompts_v5/`：V5 的主体目录
- `intent_extraction.md`：意图抽取提示词
- `result_explanation.md`：结果解释提示词
- `skills/prometheus-v5-dialog/SKILL.md`：给模型补充问诊约束、语义映射和抽取规则
- `READ_V5.md`：V5 的设计说明

这个目录结构的核心意思很明确：

- V5 是一套“提示词资产包”
- skill 也是 V5 资产的一部分
- 不再单独维护 `skills_v5/`
- 不再单独维护 `prometheus_agent_v5/`

## 3. 每个文件的作用

### 3.1 `prompts_v5/intent_extraction.md`

作用：

- 把用户自然语言问题抽取成结构化 JSON
- 只做信息识别，不做 PromQL 生成

主要输出字段包括：

- `prometheus_url`
- `job`
- `instance`
- `intent_type`
- `semantic_hint`
- `range_hours`
- `step_seconds`
- `current_window`
- `comparison`
- `threshold`
- `top_n`

这个提示词适合放在平台的第一步。

### 3.2 `prompts_v5/skills/prometheus-v5-dialog/SKILL.md`

作用：

- 约束模型不要自由发挥
- 告诉模型有哪些固定语义类别
- 告诉模型常见中文问法应该映射到什么结构化意图

它本质上不是执行逻辑，而是模型的行为边界说明。

### 3.3 `prompts_v5/result_explanation.md`

作用：

- 在 Python 工具已经返回确定性结果之后
- 让模型生成一段短结论和建议

这个提示词是可选的。

如果平台只想直接展示 Python 返回的表格结果，可以不调用它。

## 4. V5 的职责边界

V5 负责：

- 理解“用户想查什么”
- 识别目标 Prometheus 地址
- 识别 job 类型
- 识别筛选条件、阈值、时间范围
- 输出稳定 JSON

V5 不负责：

- 直接生成 PromQL
- 直接查询 Prometheus
- 维护执行计划
- 存储批次数据
- 生成 HTML 巡检报告

这些事情都应该继续放在执行层工具里。

## 5. 推荐调用链路

推荐把平台编排成下面这样：

```text
用户输入自然语言问题
  -> 大模型 + prompts_v5/intent_extraction.md
  -> 同时挂载 prompts_v5/skills/prometheus-v5-dialog/SKILL.md
  -> 输出结构化意图 JSON
  -> 平台调用现有 Python 工具执行查询 / 巡检 / 条件筛选
  -> 如有需要，再调用 prompts_v5/result_explanation.md
  -> 返回最终结果给用户
```

## 6. 三类典型问法

### 6.1 巡检类

例如：

```text
帮我看下 10.22.23.24 上的 redis 有没有异常
```

适合抽成：

- `intent_type = inspection`
- `job = redis`
- `semantic_hint = redis-health`

然后交给现有巡检型 Python 工具执行。

### 6.2 条件筛选类

例如：

```text
帮我查询下内存使用率超过了 70% 的服务器
```

适合抽成：

- `intent_type = metric_filter`
- `job = node`
- `semantic_hint = node-memory-usage`
- `comparison = >`
- `threshold = 70`

然后交给条件查询工具执行。

### 6.3 排名类

例如：

```text
CPU 最高的 10 台服务器
```

适合抽成：

- `intent_type = metric_topn`
- `job = node`
- `semantic_hint = node-cpu-top`
- `top_n = 10`

然后交给排序查询工具执行。

## 7. 为什么这样拆更合理

原因很简单：

1. 你们平台已经负责流程编排
2. 大模型能力偏弱，不适合直接自由生成 PromQL
3. 临时问诊最关键的是“意图稳定抽取”，不是“再写一套后端服务”
4. skill 和 prompt 更适合沉淀问诊经验、映射规则和输出约束

所以 V5 更适合定义成：

- 一套提示词
- 一个问诊 skill
- 一套面向平台编排的规范

而不是一套新的服务代码。

## 8. 后续扩展方式

如果后面要继续扩展 V5，建议优先扩的是下面这些内容：

1. 扩展 `intent_extraction.md` 的样例
2. 扩展 `SKILL.md` 里的语义类别
3. 扩展更多常见问法的中文别名
4. 让执行层 Python 工具支持更多固定查询模板

不要优先做的事情是：

- 让模型自由写 PromQL
- 把任意开放式查询都交给大模型生成

那样会把稳定性拉低。

## 9. 当前结论

当前 V5 的目录和定位，应当理解为：

- `V4`：固定流程巡检 + 文件批次 + HTML 报告
- `V5`：自然语言临时问诊的提示词和 skill 资产

这两个版本是并列关系，不要混在一起。

你这个场景，其实非常适合引入 Agent，但前提是思路要对。
核心判断是：你们 OnCall 的痛点，通常不是“不会处理问题”，而是“信息分散、判断链条长、重复劳动多、跨系统切换太重”。所以 Agent 最适合先做的，不是替人拍板，也不是一上来自动改系统，而是先把运维同事从“找资料、拼上下文、跑标准检查、整理沟通信息”这些高频杂事里解放出来。
核心思路
先把 Agent 定位成 OnCall Copilot，而不是 Autonomous Operator。
它的目标应该是 4 件事：
自动理解 case：这单子更像用户不熟流程、系统报错、配置问题、权限问题，还是跨部门沟通问题。
自动拉上下文：把日志、相关代码、DB 记录、需求文档、历史 case、runbook 一次性聚合给工程师。
自动给排查路径：告诉运维“先看哪几个点、先问谁、先查哪张表、哪些现象最像历史哪个 case”。
自动沉淀经验：每个 case 处理完，反向形成知识资产，而不是靠人脑记忆。
所以第一阶段最有价值的，不是“让 Agent 解决问题”，而是“让 Agent 帮人更快进入问题”。
建议的落地顺序
我建议你们按 3 个阶段推进。
第一阶段：知识助手
先做一个能“查得到”的 Agent。
接入这些数据源：
历史 case 单
SOP / runbook
需求文档 / 流程说明
常见报错码说明
系统日志检索入口
DB schema 和常用查询模板
代码仓库的只读检索能力
这阶段 Agent 的能力是：
用户贴一个 case 编号或报错信息
Agent 自动总结问题
找出相似历史 case
给出可能原因排序
输出建议排查清单
草拟回复给产线或其他团队的话术
这一阶段就已经能明显降低新人上手难度，也能减少 senior 被反复打断。
第二阶段：工具型 Agent
让 Agent 不只是“回答”，而是能“帮你做检查”。
例如给它受控工具权限：
查日志
查指定 DB 只读信息
查工单系统
查配置中心
查发布记录
查某条流程的状态流转
对照需求文档解释当前卡控逻辑
这时它的输出不再只是建议，而是：
“我已经查了 A/B/C，发现这单最像 X”
“这个 lot 卡在步骤 Y，原因疑似是规则 Z 未满足”
“最近 24 小时同类 case 增长了 3 倍，可能不是个例”
“和上周版本发布存在时间重合，建议优先确认变更影响”
这一步是最容易真正提升 MTTA/MTTR 的。
第三阶段：半自动处置
只有当你们前两阶段的数据质量和流程边界足够清楚，才考虑让 Agent 做受控动作。
例如：
自动生成标准 SQL / 查询脚本，人工确认后执行
自动触发低风险诊断脚本
自动通知对应 owner
自动补充 case 必填信息
自动创建跨团队协作单
一开始不要让 Agent 直接改配置、改数据、重跑关键流程。半导体产线场景里，误动作成本太高，必须坚持 human-in-the-loop。
最值得优先做的 5 类场景
如果你问“第一批从哪里下手”，我会优先挑这些：
相似 case 检索
价值最大，因为你们很多经验其实已经存在，只是找不到。

报错解释 + 排查路径推荐
把 senior 的思路结构化给所有人复用。

跨系统上下文聚合
把 case、日志、DB、代码、文档串起来，减少切屏和信息遗漏。

标准化沟通辅助
自动生成给产线、开发、产品、其他支持团队的沟通摘要和追问清单。

结案沉淀
每个 case 自动抽取“现象-原因-处理-验证-预防”，形成知识库闭环。

你们要先补的基础，不是模型，而是“知识和流程结构化”
很多团队做 Agent 失败，不是模型不行，而是底层材料太乱。你们至少要整理出下面这些东西：
case 分类体系
常见问题的标准排查路径
各系统的只读查询接口
历史 case 的质量分级
关键字段统一命名
哪些动作允许 Agent 做，哪些绝对不允许
升级/escalation 规则
如果这些不先整理，Agent 只会把混乱放大。
组织上要注意的一点
不要把它当成“AI 项目”，要把它当成“运维知识生产线重构”。
真正要解决的是：
经验过度依赖少数老人
case 处理质量不稳定
新人培养慢
跨部门沟通成本高
同类问题重复劳动严重
Agent 只是承载这个能力的形式。
衡量是否做对了
建议盯这几个指标，而不是只看“回答像不像”：
首次响应时间是否下降
平均定位时间是否下降
同类 case 的重复处理时间是否下降
新人独立处理率是否上升
senior 介入比例是否下降
Agent 建议的采纳率和误导率
case 结案知识沉淀率
一句话总结
你们最适合的切入点，不是“让 Agent 代替 OnCall”，而是先做一个懂你们流程、能跨日志/DB/代码/文档/历史 case 拉通上下文的 OnCall Copilot。先辅助判断，再辅助操作，最后才考虑有限自动化。

把这些资源串起来后，OnCall Agent 的处理流程应该是什么
我建议你们的 Agent 流程设计成下面这 10 步。
1. Case 接单与结构化
Agent 先把报案内容整理成结构化输入。
要做的事：
提取 case 编号
提取 lot、机台、站点、用户、系统模块、报错码、时间
识别用户动作，比如 Track In、Track Out、Hold、Release、Recipe Download
识别环境和厂别
输入来源：
工单内容
用户原始描述
报错截图文字
输出结果：
一个标准化 case 对象
这一步的意义是：后面查日志、查 DB、查代码、查文档，都依赖这些实体。
2. Case 初分类
Agent 先判断这单大概属于哪一类。
比如分到：
流程卡控
规则阻挡
机台集成异常
数据异常
配置/权限异常
跨系统不一致
疑似代码 bug
用户理解偏差
这一步主要用：
历史 case
SOP
报错码知识
少量规则关键词
输出结果：
case_type
priority
建议走 fast 还是 deep
这一步的意义是：不同类型的 case，后面的取证顺序不一样。
3. 检索 SOP 和相似历史 Case
Agent 先不要急着下结论，先看“以前怎么处理”。
要做的事：
找最相似的历史 case
找对应模块的处理 SOP
找相同报错码的知识条目
找相关需求文档章节
这一步主要用：
知识库
SOP
历史 case
需求文档索引
输出结果：
初始排查清单
相似案例参考
可能原因候选列表
这一步的意义是：避免每次从零开始。
4. 生成排查计划
Agent 根据分类结果和 SOP，给出一套本次 case 的排查顺序。
比如：
先确认 lot 当前状态和 operation
再查用户操作时间点的 MES 日志
再验证 recipe 资格和机台绑定关系
再对照需求文档确认卡控是否设计如此
最后输出结论和建议动作
这一步主要用：
SOP
case 类型
相似历史 case
输出结果：
本单专属排查计划
这一步的意义是：让 OnCall 同事少想“下一步查什么”。
5. 并行取证
这是最核心的一步。Agent 开始从不同资源拉证据。
我建议至少拆成 4 个取证方向。
A. 日志取证
查用户操作时间点
查报错栈
查接口请求和响应
查 EAP/设备交互日志
查超时、重试、失败点
B. DB取证
查 lot 当前状态
查 route / operation
查 hold / release / rework 标记
查机台/recipe/权限/配置主数据
查最近写入是否成功
C. 代码取证
定位报错码
定位当前卡控逻辑
找判断条件
看某字段的来源
看最近提交是否改过这里
D. 文档取证
查需求文档里该功能如何设计
查迭代说明里最近变更了什么
查 SOP 是否定义了特殊处理路径
输出结果：
每条证据都要结构化，不是丢一堆原文给人看
比如要整理成：
证据来源
证据摘要
支持什么判断
可信度
原始链接/原始记录位置
6. 证据归并与冲突检查
Agent 不能只“搜到了很多东西”，还要判断这些东西是不是一致。
要做的事：
日志和 DB 是否一致
DB 状态和页面现象是否一致
系统行为和需求设计是否一致
历史 SOP 和当前版本是否冲突
多条证据之间有没有互相打架
输出结果：
证据一致性结论
冲突点列表
这一步很重要，因为很多 MES case 不是“没证据”，而是“证据互相冲突”。
7. 根因判断
到这一步，Agent 才开始做 RCA。
它要回答的核心问题是：
这是 设计内阻挡 还是 真正异常
这是 数据问题 还是 代码问题
这是 配置问题 还是 权限问题
这是 MES单系统问题 还是 跨系统同步问题
这是 用户操作偏差 还是 系统提示不清晰
理想输出应该是：
最可能根因
备选根因 1、2
每个根因背后的支持证据
置信度
比如：
根因 1：recipe 未完成该产品资格认证，属规则阻挡
根因 2：EAP 未回传最新状态，导致 MES 误判可用性
根因 3：新版本对 Track In 增加前置校验，但产线未同步流程
8. 生成处置建议
Agent 在这里不要直接执行高风险动作，而是生成分层建议。
建议分成两类：
A. 立即止血建议
先换已认证 recipe
先切到可用机台
先让工程师 release hold
先补齐缺失主数据
先回退最近变更
B. 后续修复建议
修配置
补需求说明
优化提示文案
补自动校验
修代码 bug
增加监控和告警
输出结果：
建议动作列表
风险提示
哪些动作必须人工确认
这一步一定要坚持：
第一版只做建议，不做自动写库和自动改配置
9. 生成沟通内容
这个很适合 Agent 做，而且价值很高。
它可以自动生成 3 类话术：
给产线用户的说明
给开发团队的技术摘要
给跨部门协作人的追问清单
比如给产线的说明要偏业务：
当前 lot 被系统卡控的原因
是否属规则正常阻挡
临时可行处理方式
给开发的说明要偏技术：
涉及模块
日志关键点
代码判断点
DB 关键记录
是否疑似 bug
这一步能明显减少 OnCall 同事的沟通负担。
10. 结案沉淀
最后一步不是“工单关掉”，而是“把经验喂回知识库”。
Agent 要自动抽取：
现象
影响范围
根因
处理动作
验证方式
是否属于设计如此
是否需要预防措施
适用系统/模块/厂别/版本
然后沉淀成：
历史 case
标准知识条目
SOP 补充建议
规则/文档缺口清单
这一步决定了你们系统会不会越用越聪明。
